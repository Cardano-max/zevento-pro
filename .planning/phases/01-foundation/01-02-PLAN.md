    **Part A: Redis Module**

    1. **redis.service.ts** — Injectable service wrapping ioredis. Constructor connects to REDIS_URL from env. Expose methods: `get(key)`, `set(key, value, ttlSeconds)`, `del(key)`, `incr(key)`, `expire(key, ttlSeconds)`, `ttl(key)`. Handle connection errors gracefully with logging.

    2. **redis.module.ts** — Global NestJS module providing and exporting RedisService.

    **Part B: OTP Service**

    1. **otp.service.ts** — Injectable service for OTP lifecycle:
       - `generateOtp()`: Generate cryptographically random 6-digit OTP using `crypto.randomInt(100000, 999999)`.
       - `storeOtp(phone, otp)`: Hash OTP with SHA-256 (`crypto.createHash('sha256').update(otp).digest('hex')`), store in Redis with key `otp:{phone}` and 10-minute TTL. Also store attempt counter key `otp:attempts:{phone}` if not exists.
       - `verifyOtp(phone, submittedOtp)`: Retrieve hashed OTP from Redis, compare SHA-256 hash of submitted OTP. On match: delete OTP key from Redis, return true. On mismatch: increment failed attempt counter `otp:failed:{phone}`, return false. Used OTPs cannot be replayed (deleted on first successful verify).
       - Never store raw OTPs — always hashed.

    2. **msg91.service.ts** — Injectable service for MSG91 API integration:
       - `sendOtp(phone, otp)`: POST to MSG91 API endpoint to send OTP SMS. Use `MSG91_AUTH_KEY`, `MSG91_TEMPLATE_ID`, `MSG91_SENDER_ID` from env.
       - In development mode (NODE_ENV=development), log OTP to console instead of calling MSG91 API. This allows testing without burning SMS credits.
       - Handle MSG91 API errors gracefully — log error, throw HttpException with user-friendly message "Unable to send OTP. Please try again."
       - Use the MSG91 Send OTP API: `POST https://control.msg91.com/api/v5/otp` with headers `authkey: {MSG91_AUTH_KEY}` and body `{ mobile: phone, template_id: TEMPLATE_ID, otp: otp }`.

    **Part C: Rate Limit Service**

    1. **rate-limit.service.ts** — Injectable service for OTP rate limiting:
       - `checkSendLimit(phone)`: Max 5 OTP send requests per phone per hour. Uses Redis key `ratelimit:send:{phone}` with INCR + 1-hour TTL on first set. Returns `{ allowed: boolean, remainingAttempts: number, retryAfterSeconds: number }`.
       - `checkVerifyLimit(phone)`: Max 5 failed OTP verification attempts per phone per hour. Uses Redis key `ratelimit:verify:{phone}`. On 5th failure, block the phone. Returns same shape as above.
       - `resetOnSuccess(phone)`: Reset verify attempt counter on successful verification (delete `ratelimit:verify:{phone}`).
       - Rate limit window is 1 hour (3600 seconds TTL). After window expires, counter resets automatically.

    **Part D: Auth Controller and Service**

    1. **dto/send-otp.dto.ts** — `{ phone: string }` validated with class-validator: `@IsString()`, `@Matches(/^\+91[0-9]{10}$/)` (Indian phone format with country code).

    2. **dto/verify-otp.dto.ts** — `{ phone: string, otp: string, role?: string }`. Phone validated same as above. OTP: `@IsString()`, `@Length(6, 6)`. Role is optional — if user has multiple roles, they specify which to activate. Default to CUSTOMER if not specified and user has that role.

    3. **auth.service.ts** — Injectable service orchestrating the auth flow:
       - `sendOtp(phone)`: Check rate limit (checkSendLimit) -> if blocked return 429 -> generate OTP -> store in Redis -> send via MSG91 -> return success with masked phone.
       - `verifyOtp(phone, otp, requestedRole?)`: Check rate limit (checkVerifyLimit) -> if blocked return 429 -> verify OTP -> on failure increment counter and return 401 -> on success: find or create User by phone -> load user's roles -> if user has no roles, create CUSTOMER role -> if requestedRole specified, verify user has that role -> generate JWT with { userId, phone, activeRole } -> reset verify counter -> return { accessToken, user: { id, phone, name, roles, activeRole } }.
       - JWT generation: use @nestjs/jwt with JWT_SECRET from env, 7-day expiry (long-lived for mobile — per research "remember for 30 days" pattern, but start with 7 days).

    4. **auth.controller.ts** — Two endpoints:
       - `POST /auth/otp/send` — accepts SendOtpDto, calls authService.sendOtp. Returns `{ message: "OTP sent", phone: "****1234" }` (masked).
       - `POST /auth/otp/verify` — accepts VerifyOtpDto, calls authService.verifyOtp. Returns `{ accessToken, user }`.
       Both endpoints are public (no auth guard).

    5. **auth.module.ts** — NestJS module importing RedisModule, PrismaModule, JwtModule (register with secret from env, signOptions: { expiresIn: '7d' }). Provides all auth services. Exports JwtModule for use by guards.

    Install additional dependencies: `@nestjs/jwt`, `@nestjs/passport`, `passport`, `passport-jwt`, `class-validator`, `class-transformer`, `@types/passport-jwt`.

    **Part A: JWT Strategy and Guards**

    1. **interfaces/jwt-payload.interface.ts** — Define `JwtPayload` interface: `{ userId: string, phone: string, activeRole: string, iat?: number, exp?: number }`.

    2. **strategies/jwt.strategy.ts** — Passport JWT strategy:
       - Extract JWT from Authorization Bearer header.
       - Validate: look up user in database by userId from payload, verify user.isActive is true, verify activeRole exists in user's roles. Return user object with roles attached.
       - If user not found or inactive, throw UnauthorizedException.

    3. **guards/jwt-auth.guard.ts** — Extends `AuthGuard('jwt')` from @nestjs/passport. Used on all protected routes.

    4. **guards/roles.guard.ts** — Custom CanActivate guard:
       - Reads required roles from handler metadata (set by @Roles decorator).
       - If no roles metadata, allow access (public).
       - Checks if user's activeRole is in the required roles list.
       - Returns true if match, false otherwise (throws ForbiddenException).

    5. **decorators/roles.decorator.ts** — `@Roles(...roles: string[])` sets metadata key 'roles' on the handler using `SetMetadata`.

    6. **decorators/current-user.decorator.ts** — `@CurrentUser()` parameter decorator extracting `request.user` from the execution context. Usage: `@CurrentUser() user: User` on controller methods.

    Update **auth.module.ts**: add PassportModule.register({ defaultStrategy: 'jwt' }), register JwtStrategy as a provider, export PassportModule and JwtModule.

    **Part B: Admin Role Management**

    1. **dto/manage-role.dto.ts** — `AssignRoleDto`: `{ role: string, contextId?: string }`. Role validated with `@IsIn(['CUSTOMER', 'PLANNER', 'SUPPLIER', 'ADMIN'])`.

    2. **admin.service.ts** — Injectable service:
       - `assignRole(userId, role, contextId?, grantedBy)`: Check if user exists -> check if role already assigned (and active) -> create UserRole record with grantedBy set to admin's userId -> return updated user with roles.
       - `revokeRole(userId, roleId, revokedBy)`: Find UserRole by id -> verify it belongs to the specified user -> set isActive=false, revokedAt=now() -> return updated user. Do NOT hard delete — soft revoke for audit trail.
       - `listUsers(page, limit, roleFilter?)`: Paginated user list with role filter. Include user's active roles in response.
       - `getUser(userId)`: Get single user with all roles.

    3. **admin.controller.ts** — All endpoints guarded with `@UseGuards(JwtAuthGuard, RolesGuard)` and `@Roles('ADMIN')`:
       - `POST /admin/users/:id/roles` — accepts AssignRoleDto, calls adminService.assignRole. Returns updated user.
       - `DELETE /admin/users/:id/roles/:roleId` — calls adminService.revokeRole. Returns updated user.
       - `GET /admin/users` — calls adminService.listUsers with query params `page`, `limit`, `role`.
       - `GET /admin/users/:id` — calls adminService.getUser.

    4. **admin.module.ts** — NestJS module importing PrismaModule and AuthModule. Provides AdminService, AdminController.
