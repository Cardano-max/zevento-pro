<objective>
Build the privacy infrastructure: consent tracking service, contact masking logic, phone number masking interceptor, consent-required guard, and GDPR/DPDP-style audit log — delivering requirements PRIV-01 through PRIV-04.

Purpose: Customer phone numbers must never leak to vendors without explicit consent. The India DPDP Act requires consent records with timestamps, IP, and purpose. This infrastructure is foundational — every future module that exposes user contact data must use these services. Building it now prevents privacy debt from accumulating across later phases.

Output: ConsentService for tracking consent grants/revocations, ContactMaskingService for phone masking, MaskPhoneInterceptor for automatic response masking, ConsentRequiredGuard for phone-reveal endpoints, AuditLogService for admin compliance queries.
</objective>

    **Part B: Consent Service**

    1. **consent.service.ts** — Injectable service for consent lifecycle:
       - `grantConsent(params: { userId, consentType, targetUserId?, ipAddress, userAgent, metadata? })`: Create ConsentLog record with status GRANTED. The `metadata` field stores context like `{ targetUserId: "vendor-uuid", leadId: "lead-uuid" }` for PHONE_REVEAL consent. Return the created consent record.
       - `revokeConsent(params: { userId, consentType, targetUserId?, ipAddress, userAgent })`: Create a NEW ConsentLog record with status REVOKED (append-only log — never update existing records). Return the revocation record.
       - `hasActiveConsent(userId, consentType, targetUserId?)`: Query ConsentLog for the most recent record matching userId + consentType + targetUserId (from metadata). Return true if latest record status is GRANTED, false if REVOKED or no record exists.
       - `getConsentHistory(userId, consentType?)`: Return all consent records for a user, optionally filtered by type, ordered by createdAt DESC. For admin audit purposes.
       - `hasPhoneRevealConsent(customerId, vendorId)`: Convenience method — calls hasActiveConsent with PHONE_REVEAL type and targetUserId=vendorId in metadata check.

    2. **dto/grant-consent.dto.ts** — Validated DTO:
       ```typescript
       {
         consentType: string;    // @IsIn(['PHONE_REVEAL', 'LEAD_CREATION', 'DATA_PROCESSING'])
         targetUserId?: string;  // @IsUUID() @IsOptional() — the vendor who will see the phone
         metadata?: Record<string, any>; // @IsOptional()
       }
       ```

    3. **dto/revoke-consent.dto.ts** — Validated DTO:
       ```typescript
       {
         consentType: string;    // @IsIn(['PHONE_REVEAL', 'LEAD_CREATION', 'DATA_PROCESSING'])
         targetUserId?: string;  // @IsUUID() @IsOptional()
       }
       ```

    4. **consent.controller.ts** — Endpoints (require JWT auth):
       - `POST /privacy/consent` — Grant consent. Extracts IP from request, user agent from headers. Calls consentService.grantConsent. Returns consent record.
       - `DELETE /privacy/consent` — Revoke consent. Accepts RevokeConsentDto in body. Calls consentService.revokeConsent. Returns revocation record.
       - `GET /privacy/consent/status?type=PHONE_REVEAL&targetUserId=xxx` — Check if active consent exists. Returns `{ hasConsent: boolean, grantedAt?: Date }`.
       - `GET /privacy/consent/history` — Get consent history for authenticated user. Returns paginated list.
       All endpoints require @UseGuards(JwtAuthGuard).

    **Part C: Contact Masking Service**

    1. **contact-masking.service.ts** — Injectable service:
       - `maskPhone(phone: string): string` — Returns masked phone: `"****" + phone.slice(-4)`. Example: "+919876543210" -> "****3210".
       - `maskEmail(email: string): string` — Returns masked email: first 2 chars + "****" + "@" + domain. Example: "user@gmail.com" -> "us****@gmail.com".
       - `maskUserData(userData: any, fieldsToMask: string[]): any` — Deep clone the object, apply masking to specified fields. Handles nested objects. Used by the interceptor.
       - `shouldMaskForRole(viewerRole: string, dataOwnerRole: string): boolean` — Returns true if masking is needed. Rules:
         - ADMIN sees everything unmasked
         - CUSTOMER data is masked for PLANNER and SUPPLIER roles unless consent exists
         - PLANNER/SUPPLIER data is visible to ADMIN only

    **Part D: Phone Masking Interceptor**

    1. **interceptors/mask-phone.interceptor.ts** — NestJS interceptor:
       - Applied to specific controllers/routes that return user data with phone numbers.
       - Intercepts the response after the handler executes.
       - Checks if the requesting user's activeRole should see masked data (using ContactMaskingService.shouldMaskForRole).
       - If masking required, scans the response object for `phone` and `email` fields and applies masking.
       - Does NOT mask if the viewer has active PHONE_REVEAL consent for the specific data owner (checks via ConsentService).
       - Uses NestJS `CallHandler` and `Observable` pattern: `return next.handle().pipe(map(data => this.maskFields(data, request.user)))`.

    **Part A: Consent-Required Guard**

    1. **decorators/requires-consent.decorator.ts** — Custom decorator `@RequiresConsent(consentType: string)`:
       - Sets metadata on the handler: `{ consentType, targetUserIdParam }`.
       - `targetUserIdParam` specifies which route/query param contains the target user ID (default: 'userId').
       - Usage example: `@RequiresConsent('PHONE_REVEAL')` on an endpoint that reveals a customer's phone.

    2. **guards/consent-required.guard.ts** — NestJS CanActivate guard:
       - Reads consent metadata from handler (set by @RequiresConsent decorator).
       - If no consent metadata, allow access (not a consent-gated endpoint).
       - Extracts the target user ID from request params/query using the `targetUserIdParam` key.
       - Calls ConsentService.hasActiveConsent(targetUserId, consentType, viewerUserId).
       - ADMIN role bypasses consent check (admins can view all data for compliance).
       - If consent exists: allow access, log the access event.
       - If no consent: throw ForbiddenException with message "Consent required to access this data. Please request consent first."
       - On every access (allowed or denied), create an audit log entry via AuditLogService.

    **Part B: Audit Log Service**

    1. **audit-log.service.ts** — Injectable service for compliance auditing:
       - `logContactReveal(params: { viewerUserId, viewerRole, targetUserId, targetField, accessGranted, ipAddress, timestamp })`: Creates a ConsentLog record with consentType='PHONE_REVEAL' and metadata containing the reveal details. This is the compliance trail — even denied access attempts are logged.
       - `getAuditTrail(filters: { userId?, targetUserId?, consentType?, dateFrom?, dateTo?, page, limit })`: Paginated query of ConsentLog with filters. Returns records sorted by createdAt DESC. Used by admin panel for compliance auditing.
       - `getRevealHistory(targetUserId)`: Get all PHONE_REVEAL events for a specific customer. Returns list of who viewed their phone, when, and whether access was granted. For customer's own data access request (DPDP Act right).
       - `exportAuditLog(filters, format: 'json')`: Export filtered audit trail as JSON array. For compliance reporting.

    **Part C: Update consent controller with admin audit endpoints**

    Add to consent.controller.ts (guarded with @Roles('ADMIN')):
    - `GET /privacy/audit-trail` — Admin endpoint. Query params: userId, targetUserId, consentType, dateFrom, dateTo, page, limit. Returns paginated audit trail.
    - `GET /privacy/audit-trail/user/:userId` — Admin endpoint. Returns full consent/reveal history for a specific user.
    - `GET /privacy/audit-trail/reveals` — Admin endpoint. Returns all PHONE_REVEAL events. This is the critical report — shows every time a vendor saw a customer's phone number.
