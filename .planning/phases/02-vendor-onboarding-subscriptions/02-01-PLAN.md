2. **Create DTOs** with class-validator decorators:

   - `create-profile.dto.ts` (Step 2): `businessName` (IsString, MinLength 2, MaxLength 200), `categoryIds` (IsArray, IsUUID each, ArrayMinSize 1), `description` (IsString, IsOptional, MaxLength 2000), `pricingMin` (IsInt, IsOptional, Min 0 — amount in paise), `pricingMax` (IsInt, IsOptional, Min 0). Validate pricingMax >= pricingMin when both present (custom validator or service-level check).

   - `update-portfolio.dto.ts` (Step 3 metadata): `caption` (IsString, IsOptional, MaxLength 255), `categoryId` (IsUUID, IsOptional), `sortOrder` (IsInt, IsOptional, Min 0). Photo file comes via FileInterceptor, not DTO.

   - `update-service-area.dto.ts` (Step 4): `serviceAreas` (IsArray, ArrayMinSize 1) where each element has `marketId` (IsUUID) and `radiusKm` (IsInt, Min 5, Max 100, default 25). Use ValidateNested + Type decorator.

   - `submit-kyc.dto.ts` (Step 5 metadata): `documentType` (IsEnum of KycDocumentType). Document file comes via FileInterceptor.

3. **Create `vendor-owner.guard.ts`**: CanActivate guard that checks the authenticated user owns the vendor profile being accessed. Extract vendorId from route params, look up VendorProfile, compare userId with req.user.userId. ADMIN role bypasses this check. Throw ForbiddenException if not owner.

4. **Create `vendor.service.ts`** with methods:

   - `createOrGetProfile(userId: string, role: string)`: Find existing VendorProfile for user, or create one with status DRAFT, onboardingStep 1. Return profile. Called after auth — Step 1 (phone+name+role) is handled by Phase 1 OTP auth.

   - `updateBusinessDetails(vendorId: string, dto: CreateProfileDto)`: Validate all categoryIds exist in EventCategory table. Upsert VendorCategory records (delete old, insert new). Update businessName, description, pricingMin, pricingMax. Set onboardingStep = max(current, 2). Return updated profile.

   - `uploadPhoto(vendorId: string, file: Express.Multer.File, dto: UpdatePortfolioDto)`: Upload to Cloudinary with folder `vendors/${vendorId}/portfolio`. Create PortfolioPhoto record with cloudinaryPublicId, cloudinaryUrl, caption, categoryId, sortOrder. Set onboardingStep = max(current, 3). Return photo record. Max 20 photos per vendor (count check before upload).

   - `deletePhoto(vendorId: string, photoId: string)`: Find photo, verify belongs to vendor, delete from Cloudinary, delete from DB. Return void.

   - `getPhotos(vendorId: string)`: Return all PortfolioPhoto records for vendor, ordered by sortOrder.

   - `updateServiceAreas(vendorId: string, dto: UpdateServiceAreaDto)`: Validate all marketIds exist in Market table and have status ACTIVE. Delete existing VendorServiceArea records for vendor. Insert new ones. Set onboardingStep = max(current, 4). Return updated areas with market data.

   - `uploadKycDocument(vendorId: string, file: Express.Multer.File, dto: SubmitKycDto)`: Validate document type is appropriate for vendor role (Planner: AADHAAR or PAN; Supplier: GST_CERTIFICATE required, plus AADHAAR or PAN). Upload to Cloudinary with folder `vendors/${vendorId}/kyc`. Create KycDocument record. Return document record.

   - `deleteKycDocument(vendorId: string, docId: string)`: Find doc, verify belongs to vendor, delete from Cloudinary, delete from DB.

   - `submitForKyc(vendorId: string)`: Validate onboardingStep >= 4 (all previous steps complete). Validate at least 1 KYC document uploaded. For SUPPLIER role, validate GST_CERTIFICATE is present. Set status = PENDING_KYC, onboardingStep = 5, submittedAt = now(). Create AdminNotification record (type: KYC_SUBMISSION, title: "New KYC submission", message with vendor business name, referenceId: vendorId). Return updated profile.

   - `getProfile(vendorId: string)`: Return full profile with categories, photos, serviceAreas (with market data), kycDocuments, subscription.

   - `getMyProfile(userId: string)`: Find VendorProfile by userId, return with all relations.

   All methods use PrismaService. Throw NotFoundException for missing records, BadRequestException for validation failures, ConflictException for duplicate submissions.

5. **Create `vendor.controller.ts`** with routes (all protected by JwtAuthGuard + RolesGuard with PLANNER or SUPPLIER roles):

   - `POST /vendor/profile` — createOrGetProfile. Uses @CurrentUser to get userId and activeRole.
   - `PATCH /vendor/profile/business` — updateBusinessDetails. Uses VendorOwnerGuard.
   - `POST /vendor/profile/photos` — uploadPhoto. Uses @UseInterceptors(FileInterceptor('photo', { storage: memoryStorage(), limits: { fileSize: 10 * 1024 * 1024 }, fileFilter: imageOnly })) where imageFilter rejects non-image mimetypes. Uses VendorOwnerGuard.
   - `GET /vendor/profile/photos` — getPhotos.
   - `DELETE /vendor/profile/photos/:photoId` — deletePhoto. Uses VendorOwnerGuard.
   - `PUT /vendor/profile/service-areas` — updateServiceAreas. Uses VendorOwnerGuard.
   - `POST /vendor/profile/kyc/documents` — uploadKycDocument. Uses FileInterceptor (same config, allows image + pdf). Uses VendorOwnerGuard.
   - `DELETE /vendor/profile/kyc/documents/:docId` — deleteKycDocument. Uses VendorOwnerGuard.
   - `POST /vendor/profile/kyc/submit` — submitForKyc. Uses VendorOwnerGuard.
   - `GET /vendor/profile/me` — getMyProfile. Returns current user's vendor profile with all relations.

6. **Create `vendor.module.ts`**: Import PrismaModule, CloudinaryModule. Provide VendorService, VendorOwnerGuard. Export VendorService.

7. **Update `app.module.ts`**: Add VendorModule to imports.

8. **Important details**:
   - The `onboardingStep` uses max(current, N) pattern — vendors can revisit earlier steps without losing progress.
   - Status transitions: DRAFT (during onboarding) -> PENDING_KYC (after submit) -> APPROVED/REJECTED (by admin in 02-03).
   - REJECTED vendors can re-edit and re-submit (status goes back to PENDING_KYC).
   - File filter for photo uploads: accept only image/jpeg, image/png, image/webp. For KYC docs: also accept application/pdf.
   - All amounts stored in paise (not rupees).
