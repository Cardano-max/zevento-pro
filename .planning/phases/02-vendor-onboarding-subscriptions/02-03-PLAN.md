1. **Create DTOs**:

   - `review-kyc.dto.ts`: `action` (IsEnum: 'APPROVE' | 'REJECT'), `rejectionReason` (IsString, IsOptional, MaxLength 500 — required when action is REJECT, validated at service level).

   - `manage-vendor.dto.ts`: `isActive` (IsBoolean, IsOptional — for suspend/reactivate toggle).

2. **Add methods to `admin.service.ts`** (extend existing service, do NOT rewrite existing methods):

   - `getKycQueue(status?: string, page: number = 1, limit: number = 20)`: Query VendorProfile with status filter (default: PENDING_KYC). Include user (phone, name), categories (with category names), serviceAreas (with market city), kycDocuments, photos count. Paginate with skip/take. Return `{ data, total, page, totalPages }`.

   - `getVendorDetail(vendorId: string)`: Return full VendorProfile with all relations: user, categories, photos, serviceAreas with markets, kycDocuments, subscription with plan. Throw NotFoundException if not found.

   - `reviewKyc(vendorId: string, dto: ReviewKycDto, adminUserId: string)`:
     1. Find VendorProfile, verify status is PENDING_KYC. Throw BadRequestException if not.
     2. If APPROVE: set status to APPROVED, approvedAt to now(), clear rejectionReason.
     3. If REJECT: validate rejectionReason is provided (throw BadRequestException if empty). Set status to REJECTED, set rejectionReason. Do NOT set approvedAt.
     4. Create AdminNotification: type KYC_SUBMISSION, title "KYC {action}d: {businessName}", referenceId vendorId.
     5. Return updated profile.

   - `suspendVendor(vendorId: string)`: Set VendorProfile status to SUSPENDED. Return updated profile.

   - `reactivateVendor(vendorId: string)`: Set VendorProfile status back to APPROVED (only if was SUSPENDED). Throw BadRequestException if current status is not SUSPENDED.

   - `listVendors(status?: string, role?: string, page: number = 1, limit: number = 20)`: Paginated vendor list with filters. Include user, subscription status. Return paginated result.

3. **Add endpoints to `admin.controller.ts`** (extend existing controller, all existing endpoints must remain):

   - `GET /admin/vendors` — listVendors. Query params: status, role, page, limit.
   - `GET /admin/vendors/kyc-queue` — getKycQueue. Query params: status (default PENDING_KYC), page, limit.
   - `GET /admin/vendors/:vendorId` — getVendorDetail.
   - `POST /admin/vendors/:vendorId/kyc-review` — reviewKyc. Body: ReviewKycDto.
   - `PATCH /admin/vendors/:vendorId/suspend` — suspendVendor.
   - `PATCH /admin/vendors/:vendorId/reactivate` — reactivateVendor.

   All endpoints use existing `@UseGuards(JwtAuthGuard, RolesGuard)` and `@Roles(Role.ADMIN)` from the existing admin controller pattern.

4. **Update `admin.module.ts`** if needed: Ensure PrismaModule is imported (should already be).

5. **Important details**:
   - KYC review is always manual (per discretion decision — early stage, low volume).
   - Rejected vendors can edit their profile and re-submit (02-01 handles the re-submit flow — status goes back to PENDING_KYC).
   - Suspended vendors: status SUSPENDED means hidden from marketplace, no leads. Different from REJECTED (can re-submit) — SUSPENDED is an admin enforcement action.
   - Existing user management endpoints (assignRole, revokeRole, listUsers) from Phase 1 remain unchanged. The new vendor endpoints complement them.

1. **Create DTOs**:

   - `manage-category.dto.ts`:
     - `CreateCategoryDto`: `name` (IsString, MinLength 2, MaxLength 100), `description` (IsString, IsOptional, MaxLength 500), `parentId` (IsUUID, IsOptional — for subcategories), `sortOrder` (IsInt, IsOptional, default 0).
     - `UpdateCategoryDto`: All fields from CreateCategoryDto but all IsOptional, plus `isActive` (IsBoolean, IsOptional).

   - `manage-plan.dto.ts`:
     - `CreatePlanDto`: `name` (IsString, MinLength 2, MaxLength 100), `vendorRole` (IsEnum: PLANNER | SUPPLIER), `tier` (IsEnum: BASIC | PREMIUM), `amountPaise` (IsInt, Min 100 — minimum 1 rupee), `periodMonths` (IsInt, Min 1, Max 12, default 1), `features` (IsOptional, IsObject — JSON feature list for display).
     - `UpdatePlanDto`: `name` (IsOptional), `amountPaise` (IsOptional, IsInt, Min 100), `features` (IsOptional), `isActive` (IsBoolean, IsOptional). NOTE: vendorRole and tier are NOT updatable (changing them would break existing subscriptions — create a new plan instead).

2. **Add methods to `admin.service.ts`**:

   **Event Categories:**
   - `createCategory(dto: CreateCategoryDto)`: Generate slug from name (lowercase, replace spaces with hyphens, remove special chars). Check name uniqueness (throw ConflictException on duplicate). If parentId provided, verify parent exists. Create and return category.
   - `updateCategory(categoryId: string, dto: UpdateCategoryDto)`: Find category (throw NotFoundException). If name changed, regenerate slug and check uniqueness. Update fields. Return updated category.
   - `listCategories(includeInactive: boolean = false)`: Return all categories ordered by sortOrder, optionally including inactive. Include parent/children relations. Include vendor count (count of VendorCategory records).
   - `getCategoryDetail(categoryId: string)`: Return category with vendor count and subcategories.

   **Subscription Plans:**
   - `createPlan(dto: CreatePlanDto)`: Check unique constraint on [vendorRole, tier] (throw ConflictException if exists). Create SubscriptionPlan. razorpayPlanId is NULL initially — gets lazily created when first vendor subscribes (handled in 02-02). Return created plan.
   - `updatePlan(planId: string, dto: UpdatePlanDto)`: Find plan (throw NotFoundException). Update allowed fields. If amountPaise changes AND plan has razorpayPlanId, set razorpayPlanId to NULL (forces re-creation in Razorpay on next checkout — Razorpay plans are immutable, price change = new plan). Return updated plan.
   - `listPlans(vendorRole?: string, includeInactive: boolean = false)`: Return plans with optional role filter and active filter. Include count of active subscriptions per plan.
   - `getPlanDetail(planId: string)`: Return plan with active subscription count.

   **Admin Notifications:**
   - `getNotifications(page: number = 1, limit: number = 20, unreadOnly: boolean = false)`: Paginated AdminNotification list, newest first. Filter by isRead if unreadOnly. Return `{ data, total, unreadCount }`.
   - `markNotificationRead(notificationId: string)`: Set isRead to true. Return updated notification.
   - `markAllNotificationsRead()`: Update all unread to isRead=true. Return count updated.
   - `getUnreadCount()`: Return count of unread notifications.

3. **Add endpoints to `admin.controller.ts`** (extending — keep all existing endpoints):

   **Event Categories:**
   - `POST /admin/categories` — createCategory
   - `PATCH /admin/categories/:categoryId` — updateCategory
   - `GET /admin/categories` — listCategories. Query: includeInactive (boolean).
   - `GET /admin/categories/:categoryId` — getCategoryDetail

   **Subscription Plans:**
   - `POST /admin/subscription-plans` — createPlan
   - `PATCH /admin/subscription-plans/:planId` — updatePlan
   - `GET /admin/subscription-plans` — listPlans. Query: vendorRole, includeInactive.
   - `GET /admin/subscription-plans/:planId` — getPlanDetail

   **Notifications:**
   - `GET /admin/notifications` — getNotifications. Query: page, limit, unreadOnly.
   - `GET /admin/notifications/unread-count` — getUnreadCount.
   - `PATCH /admin/notifications/:notificationId/read` — markNotificationRead.
   - `POST /admin/notifications/mark-all-read` — markAllNotificationsRead.

   All endpoints protected with ADMIN role guard.

4. **Important details**:
   - Slug generation for categories: `name.toLowerCase().replace(/[^a-z0-9]+/g, '-').replace(/^-|-$/g, '')`. If slug collision after update, append a numeric suffix.
   - Disabling a category (isActive=false) does NOT remove it from existing vendor profiles — it just hides it from future selection. This preserves historical data.
   - Disabling a subscription plan (isActive=false) does NOT cancel existing subscriptions — it just prevents new signups on that plan.
   - When plan price changes and razorpayPlanId is reset, existing subscribers continue on the old Razorpay plan. Only new checkouts get the new price. This is correct behavior — price changes should not retroactively affect existing subscriptions.
   - Admin notifications are lightweight — no push/email for now. The admin panel (Phase 4 UI) will poll `GET /admin/notifications/unread-count` to show a badge.
