  <action>
1. **Install Razorpay SDK**: `cd api && pnpm add razorpay`

2. **Create `razorpay.service.ts`**: Singleton injectable wrapping the Razorpay SDK instance.
   - Constructor: Initialize `new Razorpay({ key_id: config.get('RAZORPAY_KEY_ID'), key_secret: config.get('RAZORPAY_KEY_SECRET') })` via ConfigService.
   - In dev mode (NODE_ENV=development), if env vars not set, log warning and set a `devMode` flag. Methods return mock data when devMode is true.
   - Methods:
     - `createPlan(params: { period: string; interval: number; item: { name: string; amount: number; currency: string; description: string } })` — Calls `this.instance.plans.create(params)`. Returns Razorpay plan object with `id`.
     - `createSubscription(params: { plan_id: string; total_count: number; customer_notify?: number; notes?: Record<string, string> })` — Calls `this.instance.subscriptions.create(params)`. Returns subscription object with `id`, `short_url`, `status`.
     - `cancelSubscription(subscriptionId: string, cancelAtCycleEnd: boolean)` — Calls `this.instance.subscriptions.cancel(subscriptionId, cancelAtCycleEnd)`.
     - `fetchSubscription(subscriptionId: string)` — Calls `this.instance.subscriptions.fetch(subscriptionId)`.
     - `validateWebhookSignature(body: string, signature: string, secret: string): boolean` — Uses `Razorpay.validateWebhookSignature(body, signature, secret)`. If that method doesn't exist on the SDK, fallback to manual HMAC-SHA256: `crypto.createHmac('sha256', secret).update(body).digest('hex') === signature`.

3. **Create `subscription.service.ts`**: Injectable service with PrismaService + RazorpayService.
   - `getPlansForRole(vendorRole: string)`: Query SubscriptionPlan where vendorRole matches and isActive is true. Return plan list.
   - `getAllPlans()`: Return all active subscription plans (for admin listing).
   - `initateCheckout(vendorId: string, planId: string)`:
     1. Find VendorProfile by vendorId, verify status is APPROVED (vendors must be KYC-approved before subscribing). Throw BadRequestException if not.
     2. Find SubscriptionPlan by planId, verify isActive. Throw NotFoundException if not found.
     3. Verify plan's vendorRole matches vendor's role. Throw BadRequestException if mismatch.
     4. Check if vendor already has an ACTIVE subscription. Throw ConflictException if so.
     5. If plan has no razorpayPlanId, create plan in Razorpay first (lazy sync pattern): call `razorpayService.createPlan({ period: 'monthly', interval: plan.periodMonths, item: { name: plan.name, amount: plan.amountPaise, currency: 'INR', description: plan.name } })`. Save returned plan.id as razorpayPlanId.
     6. Create Razorpay subscription: `razorpayService.createSubscription({ plan_id: plan.razorpayPlanId, total_count: 120 (10 years of monthly), customer_notify: 1, notes: { vendorId, planId } })`.
     7. Create or update VendorSubscription record: planId, razorpaySubscriptionId, status: CREATED.
     8. Return `{ subscriptionId: razorpaySubscription.id, shortUrl: razorpaySubscription.short_url }` — vendor uses shortUrl to complete payment.
   - `getMySubscription(vendorId: string)`: Return VendorSubscription with plan details.
   - `cancelSubscription(vendorId: string)`: Find active subscription, call razorpayService.cancelSubscription with cancelAtCycleEnd=true. Update local status to CANCELLED. Return updated subscription.

4. **Create `subscription.controller.ts`**: Protected by JwtAuthGuard + RolesGuard (PLANNER, SUPPLIER).
   - `GET /subscriptions/plans` — getPlansForRole(user.activeRole). Public-ish but still auth'd to know the role.
   - `POST /subscriptions/checkout` — Body: `{ planId: string }`. Calls initiateCheckout. Returns shortUrl for payment.
   - `GET /subscriptions/me` — getMySubscription for current vendor.
   - `POST /subscriptions/cancel` — cancelSubscription for current vendor.

5. **Create `subscription.module.ts`**: Import PrismaModule, ConfigModule. Provide RazorpayService, SubscriptionService. Export SubscriptionService, RazorpayService (webhook module needs RazorpayService).

6. **Update `app.module.ts`**: Add SubscriptionModule to imports.

    - Unauthenticated requests return 401, CUSTOMER role returns 403

  <name>Task 2: Razorpay subscription webhook handler with idempotency and transaction recording</name>
  <files>
    api/src/subscription/webhook/subscription-webhook.controller.ts
    api/src/subscription/webhook/subscription-webhook.service.ts

  <action>
1. **Create `subscription-webhook.controller.ts`**: PUBLIC endpoint (no JwtAuthGuard — webhooks come from Razorpay, not authenticated users).
   - `POST /webhooks/razorpay/subscription` — Receives raw body via `@Req() req` (using rawBody from NestFactory config in 02-01). Extracts `x-razorpay-signature` header.
   - Pass `req.rawBody.toString()` (NOT parsed body) and signature to webhook service.
   - Return `{ status: 'ok' }` with 200 on success. Return 400 on signature failure. Always return 200 for already-processed events (idempotent).

2. **Create `subscription-webhook.service.ts`**: Injectable with PrismaService, RazorpayService, ConfigService.
   - `handleWebhook(rawBody: string, signature: string)`:
     1. **Verify signature**: Call `razorpayService.validateWebhookSignature(rawBody, signature, config.get('RAZORPAY_WEBHOOK_SECRET'))`. Throw UnauthorizedException on failure.
     2. **Parse payload**: `JSON.parse(rawBody)` AFTER verification. Extract `event` (string like 'subscription.charged'), `payload.subscription.entity` (subscription object), `payload.payment.entity` (payment object, present on charged events).
     3. **Idempotency check**: Try to create WebhookEvent with `provider: 'RAZORPAY'`, `externalId: payload.subscription.entity.id + '_' + event_timestamp_or_payment_id`, `eventType: event`, `status: 'RECEIVED'`, `payload: parsedPayload`. If unique constraint violation (P2002), log "duplicate event, skipping" and return. This reuses the existing webhook_events table from Phase 1.
     4. **Process by event type** — use a switch on the event string:

        - `subscription.authenticated`: Update VendorSubscription status to AUTHENTICATED.
        - `subscription.activated`: Update status to ACTIVE. Set currentPeriodStart and currentPeriodEnd from payload.
        - `subscription.charged`: Update status to ACTIVE. Update currentPeriodStart/End. Create Transaction record: type SUBSCRIPTION, amountPaise from payment.entity.amount, razorpayPaymentId from payment.entity.id, status PAID, paidAt from payment.entity.captured_at or now().
        - `subscription.pending`: Update status to PENDING.
        - `subscription.halted`: Update status to HALTED. (Payment retries exhausted — vendor loses access after grace period.)
        - `subscription.cancelled`: Update status to CANCELLED.
        - `subscription.paused`: Update status to PAUSED.
        - `subscription.resumed`: Update status to ACTIVE.
        - `subscription.completed`: Update status to COMPLETED.
        - Default: Log unknown event type, still mark webhook as PROCESSED.

     5. **Update webhook event**: Set status to PROCESSED, processedAt to now().
     6. **Error handling**: If processing fails, set webhook event status to FAILED. Log error. Still return 200 to Razorpay (they retry on non-2xx, and we don't want retries for processing bugs). But throw so NestJS logs the error.

   - **Find subscription**: Look up VendorSubscription by razorpaySubscriptionId (from payload.subscription.entity.id). If not found, log error and skip (orphaned webhook).

3. **Important details**:
   - Razorpay amounts are in paise. The `payment.entity.amount` is already in paise — store directly, no conversion needed.
   - The externalId for idempotency should be unique per event instance. Use `${subscriptionId}_${event}_${payment.entity.id || timestamp}` to differentiate between multiple charges for the same subscription.
   - Log all webhook events at info level for debugging.
   - In dev mode without Razorpay keys, the webhook endpoint should still exist but skip signature verification (log a warning instead). This allows testing the event processing flow with manual POST requests.
  </action>

    - Duplicate POST with same externalId is silently skipped (no duplicate record, no error)

    - VendorSubscription status updates match the event type
