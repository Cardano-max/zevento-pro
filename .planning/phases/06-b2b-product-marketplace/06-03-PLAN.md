const ORDER_STATUS_MESSAGES: Record<string, string> = {
  CONFIRMED: 'Your order has been confirmed by the supplier!',
  DISPATCHED: 'Your order has been dispatched and is on its way!',
  DELIVERED: 'Your order has been delivered!',
  CANCELLED: 'Your order has been cancelled.',
};
```

Inject NotificationService into OrderService constructor.

Add method `transitionOrderStatus(orderId: string, dto: TransitionOrderStatusDto, requesterId: string, requesterRole: string)`:

1. Find the order by id, include items (with product info), buyer, vendor
2. If not found, throw NotFoundException
3. **Authorization check**: Only the vendor (supplier) can transition status forward. Verify either:
   - `requesterRole === 'SUPPLIER'` and order.vendor.userId matches requesterId (the supplier owns this order)
   - `requesterRole === 'ADMIN'` (admin can always transition)
   - For CANCELLED: also allow buyer (`requesterRole === 'PLANNER' || requesterRole === 'CUSTOMER'` and order.buyerId matches requesterId) -- but only from PENDING or CONFIRMED
4. **Validate transition**: Check `VALID_ORDER_TRANSITIONS[currentStatus]?.includes(dto.status)`. If not valid, throw `BadRequestException('Invalid status transition from ${currentStatus} to ${dto.status}')`.
5. **Atomic transition** using `$transaction`:
   - `updateMany` with `where: { id: orderId, status: currentStatus }` and `data: { status: dto.status, confirmedAt/dispatchedAt/deliveredAt/cancelledAt based on status }`. If `updated.count === 0`, throw `BadRequestException('Order status conflict -- may have been updated by another request')`.
   - Create `OrderStatusHistory` record: `{ orderId, fromStatus: currentStatus, toStatus: dto.status, note: dto.note }`
   - If dto.status is CANCELLED: **restore stock** for each order item. Use `prisma.product.update({ where: { id: item.productId }, data: { stock: { increment: item.quantity } } })` inside the same transaction. This prevents stock leak (Pitfall 2 from research).
6. **After transaction** (outside $transaction, consistent with Phase 4 pattern):
   - Send push notification to the buyer: `notificationService.sendPushToCustomer(order.buyerId, { bookingId: orderId, eventType: ORDER_STATUS_MESSAGES[dto.status], status: dto.status })`. Reuse the sendPushToCustomer interface -- the field names map well enough (bookingId used as general reference ID).
   - If CANCELLED by vendor/admin: also notify buyer with cancellation message
   - If CANCELLED by buyer: also notify vendor with `notificationService.sendPushToVendor(order.vendorId, { leadId: orderId, eventType: 'Order cancelled by customer', city: 'Order Update' })`

@Patch('/orders/:id/status')
@UseGuards(JwtAuthGuard, RolesGuard)
@Roles('SUPPLIER', 'PLANNER', 'CUSTOMER', 'ADMIN')
async transitionStatus(
  @Param('id', ParseUUIDPipe) id: string,
  @Body() dto: TransitionOrderStatusDto,
  @CurrentUser() user: JwtPayload,
) {
  return this.orderService.transitionOrderStatus(
    id, dto, user.userId, user.activeRole,
  );
}
```
