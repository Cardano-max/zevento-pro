**2. Create OrderService** (`api/src/order/order.service.ts`):

Inject PrismaService, CommissionService (from PaymentModule).

Methods:

- `createOrder(buyerId: string, dto: CreateOrderDto)`: This is the core method. It must:
  1. Verify all products belong to `dto.vendorId` and are `isActive: true`
  2. Validate MOQ: each `item.quantity >= product.moq` (Open Question 5)
  3. Validate all products belong to the SAME vendor (dto.vendorId)
  4. Use `prisma.$transaction` for atomic stock reservation:
     - For each item: `findUnique` the product, check `product.stock >= item.quantity`, then `update` with `stock: { decrement: item.quantity }`
     - If any product has insufficient stock, throw BadRequestException with product name and available stock
  5. Calculate `totalPaise` server-side: `sum(item.quantity * product.pricePaise)` -- NEVER accept total from client (04-02 decision)
  6. Inside the same transaction, create `ProductOrder` with status PENDING, items as `ProductOrderItem` records (unitPaise from product.pricePaise, totalPaise = quantity * unitPaise)
  7. Create initial `OrderStatusHistory` record (fromStatus: null, toStatus: 'PENDING')
  8. After transaction succeeds, check each product's new stock against lowStockThreshold and enqueue 'stock-alerts' BullMQ jobs for any that crossed the threshold (enqueue OUTSIDE transaction, consistent with Pitfall 4 pattern from Phase 4 -- no long-running operations inside $transaction)
  9. Return the created order with items

  Inject `@InjectQueue('stock-alerts') stockAlertQueue: Queue` for low-stock alerts.

- `getOrderById(orderId: string)`: Fetch order with items (include product name, image), vendor, buyer info. Used by both planner and supplier views.

- `getMyOrders(buyerId: string, page: number, limit: number)`: Planner's order history, paginated, ordered by createdAt desc. Include vendor businessName, item count, totalPaise, status.

- `getVendorOrders(vendorId: string, page: number, limit: number, status?: string)`: Supplier's order list with optional status filter. Include buyer name, item count, totalPaise, status, createdAt.

- `cancelOrder(orderId: string, requesterId: string, requesterRole: string)`: Cancel a PENDING or CONFIRMED order. Verify requester is buyer or vendor owner. Restore stock via `prisma.product.update({ data: { stock: { increment: item.quantity } } })` for each item. Create OrderStatusHistory. **1. Extend PaymentWebhookService** (`api/src/payment/webhook/payment-webhook.service.ts`):

This is a critical modification (Pitfall 8 from research). The current `handlePaymentCaptured` method always enqueues to 'payment-processing' queue, which only handles bookings (finds Booking by razorpayOrderId).

Modify `handlePaymentCaptured` to route by `notes.type`:
```typescript
private async handlePaymentCaptured(webhookEventId: string, paymentEntity: any) {
  const notes = paymentEntity.notes || {};
  const paymentType = notes.type;

  if (paymentType === 'MARKETPLACE_SALE') {
    // Route to product order payment processing queue
    await this.productOrderPaymentQueue.add(
      'product-order-payment-captured',
      { webhookEventId, paymentEntity, orderNotes: notes },
      { attempts: 3, backoff: { type: 'exponential', delay: 60000 } },
    );
    this.logger.log(`Enqueued product-order-payment job for payment ${paymentEntity.id}`);
  } else {
    // Default: route to booking payment processing (existing behavior)
    await this.paymentQueue.add(
      'payment-captured',
      { webhookEventId, paymentEntity, orderNotes: notes },
      { attempts: 3, backoff: { type: 'exponential', delay: 60000 } },
    );
    this.logger.log(`Enqueued payment-processing job for payment ${paymentEntity.id}`);
  }
}
```

Inject `@InjectQueue('product-order-payment') private readonly productOrderPaymentQueue: Queue` in the constructor.

Also modify `handlePaymentFailed` to check `notes.type`: if MARKETPLACE_SALE, find ProductOrder by razorpayOrderId and update paymentStatus to FAILED (and restore stock via increment). If BOOKING_COMMISSION (or no type), keep existing booking behavior.

**2. Create OrderPaymentProcessor** (`api/src/order/processor/order-payment.processor.ts`):

`@Processor('product-order-payment')` BullMQ worker. Mirrors PaymentProcessor pattern exactly but for product orders.

Inject PrismaService, CommissionService, NotificationService.

`process` method:
1. Find ProductOrder by `paymentEntity.order_id` (matching on razorpayOrderId)
2. If not found, log error, mark webhook FAILED, return
3. Get commission rate: prefer locked rate on order (`order.commissionRateBps`), fallback to `commissionService.getRate(order.vendorId, null)` -- null categoryId for product orders (Pitfall 3)
4. Calculate commission split: `commissionPaise = Math.round((totalPaise * rateBps) / 10000)`, `netPayoutPaise = totalPaise - commissionPaise`
5. Create Transaction: `type: TransactionType.MARKETPLACE_SALE`, `productOrderId: order.id`, `amountPaise`, `commissionPaise`, `netPayoutPaise`, `razorpayPaymentId`, `razorpayOrderId`, `status: TransactionStatus.PAID`, `paidAt: new Date()`
6. Update ProductOrder: `paymentStatus: 'CAPTURED'`
7. Mark webhookEvent as PROCESSED
8. Send push notification to vendor: `notificationService.sendPushToVendor(order.vendorId, { leadId: order.id, eventType: 'New product order received!', city: 'Order Alert' })`

Add `@OnWorkerEvent('failed')` error handler (same pattern as PaymentProcessor).
